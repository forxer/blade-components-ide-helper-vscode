# CLAUDE.md

Guidance for Claude Code when working in this repository.

## What this is

A standalone **VS Code extension** (TypeScript/Node) that provides autocompletion (tag names,
attribute names, constrained attribute values) and hover for class-based Blade `<x-…>` components in
`.blade.php` files. It reads and merges the **VS Code Custom Data** files (`*.html-data.json`)
generated by the PHP package `forxer/blade-components-ide-helper` and its consumers (e.g.
`forxer/blade-ui-kit-bootstrap`, `axn/laravel-medialibrary-helpers`). It does **not** parse PHP —
its only input is those JSON files.

This repo is independent from the PHP packages. Published as `forxer.blade-components-ide-helper` on
the **Visual Studio Marketplace** and **Open VSX**, and attached as a `.vsix` to each GitHub Release.

## Architecture — the one rule that matters

Hard boundary between **pure logic** and **VS Code adapters**:

```
src/data/     parse + merge the *.html-data.json files      (pure)
src/context/  locate the cursor inside a <x-…> tag           (pure)
src/complete/ turn (context + data) into plain descriptors   (pure)
src/providers/ wrap descriptors into VS Code objects         (adapter)
src/vscode/   file discovery, watchers, output channel       (adapter)
src/extension.ts  activate(): wires everything together      (adapter)
```

Files under `data/`, `context/`, `complete/` must **never** `import 'vscode'`. That is what lets the
unit suite run under Mocha without launching an editor. Keep new logic pure; only adapters touch the
VS Code API.

## Gotchas (do not regress these)

- **Context parsing is quote-aware.** `findEnclosingTagStart` scans forward tracking quote state so a
  `<` or `>` inside a quoted value (e.g. `:disabled="$count > 5"`, common in Blade) does not break
  detection. `hoverTarget` likewise returns none inside a value literal.
- **The `-` is a VS Code word separator.** Completion items set an explicit `range`
  (`replaceTypedLength`) to replace the typed `x-…` fragment instead of appending.
- **`sortText`/`preselect` cannot outrank snippets.** VS Code (by design, see vscode#71660 / #79516)
  ranks input-prefix match above `sortText`. If both `.code-snippets` and this extension are present,
  snippets win the list — so the recommendation is: **when this extension is used, do not generate the
  snippets** (`blade-components-ide-helper:generate --json --ide-json`, no `--snippets`). Ranking
  tweaks only help for attribute *values*.
- **Integration tests need a GUI stack.** `@vscode/test-electron` won't launch in a bare WSL shell
  (no libnss/gtk, no xvfb). The **unit suite is the CI gate**; integration tests run in a real editor.

## Commands

```bash
npm install          # deps (like composer install)
npm run compile      # tsc → out/ (used by tests / type-check)
npm run bundle       # esbuild → dist/extension.js (the shipped file)
npm run test:unit    # compile + Mocha over the pure logic — the gate
npm run package      # produce a .vsix
```

Debug: **F5** opens the Extension Development Host (config in `.vscode/launch.json`, runs `bundle`
first). Test inside it with `test/fixtures/workspace/` (ships two real `html-data.json` +
`example.blade.php`). The file's language mode must be `blade` (a third-party Blade extension must be
installed in the host window). Note: `.vscode/` **is** versioned in this repo.

## Git workflow

Two long-lived branches:

- **`main`** — default branch, always the **latest published version**. Protected: PR + green CI only,
  releases tagged here.
- **`develop`** — the **next version**; base day-to-day work here.

External contributors fork and PR against `develop`. See `CONTRIBUTING.md` for the full dev guide.

## Releasing (maintainer)

1. On `develop`: bump `version` in `package.json` (`npm install` to sync the lock), add a
   `CHANGELOG.md` entry.
2. PR `develop → main`, merge on green CI.
3. Tag on `main`: `git tag vX.Y.Z && git push origin vX.Y.Z`. The tag triggers
   `.github/workflows/release.yml` → tests, packages, publishes to both registries, creates the GitHub
   Release. **Never create the release by hand in the UI.**

Versions are digits-only SemVer (no `-beta`; pre-releases use the publish flag). Publishing secrets:
`VSCE_PAT` (Marketplace) and `OVSX_PAT` (Open VSX), each step skipped if its secret is absent.

**Every time you prepare a release, spell out the exact commands for the user to run** — the user
drives GitHub (push, PR, tag) and may not remember the flow. After committing the version bump +
changelog on `develop`, always hand off with a block like this, filled in with the real version:

```bash
# 1. push develop
git push
# 2. open a PR develop → main titled "Release vX.Y.Z", merge once CI is green
# 3. tag the release on main (this triggers publication to both registries)
git checkout main && git pull
git tag vX.Y.Z && git push origin vX.Y.Z
```

## Working constraints

- **Never tag/release without an explicit request.** The user tests manually first.
- **Never commit without proposing the message first** (Conventional Commits) and getting approval.
- **No reference to Claude/AI** in commit messages.
- `git push` and all GitHub UI actions (settings, PRs, tags, secrets) are done **by the user**, not
  from here.
- Do **not** commit `docs/superpowers/**` (specs/plans are working documents, git-ignored).
- English only for code, comments, identifiers, and docs.
